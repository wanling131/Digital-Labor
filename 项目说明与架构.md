## 一、系统架构图

整体是**前后端分离**：浏览器里跑管理端或工人端页面，所有数据通过 HTTP 请求到后端，后端再读写本地数据库。

```mermaid
flowchart LR
  subgraph Browser["浏览器"]
    PC[管理端 PC]
    H5[工人端 H5]
  end
  subgraph API["后端 API :3000"]
    Auth[登录鉴权]
    Routes[业务路由]
  end
  subgraph DB["数据层"]
    SQLite[(SQLite labor.db)]
  end
  PC -->|"请求 + labor_token"| API
  H5 -->|"请求 + labor_worker_token"| API
  API --> Auth --> Routes
  Routes --> SQLite
```

- **管理端**：电脑访问首页后点「进入管理端」或直接访问 `/login`，账号密码登录，请求带 `labor_token`；业务页面在 `/pc/*`（如 `/pc/dashboard`）。
- **工人端**：手机或电脑访问 `/h5`，无 token 时跳 `/h5/login`，用工号+姓名登录，请求带 `labor_worker_token`。
- **后端**：Express 提供 REST 接口，校验 token 后查写 SQLite（文件在 `server/data/labor.db`）。

## 二、技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端框架 | Next.js 16 + React 19 + TypeScript | App Router，PC 与 H5 同仓 |
| 前端路由 | Next.js App Router | `/`、`/login`、`/pc/*`、`/h5/*` |
| 管理端请求 | `lib/api.ts` 的 `api()` | 自动带 token，401 跳登录 |
| 工人端请求 | `lib/api.ts` 的 `apiWorker()` | 工人 token，401 跳工人登录 |
| 前端 UI | Radix UI + Tailwind CSS | 组件在 `components/ui`、`components/pc`、`components/h5` |
| 后端运行时 | Node.js | 建议 18+ |
| 后端框架 | Express 4 | REST API |
| 数据库 | SQLite (better-sqlite3) | 单文件，无需单独装数据库 |
| 认证 | JWT (jsonwebtoken) | 7 天有效，payload 含 userId 或 workerId |
| 考勤 Excel | multer + xlsx | 上传解析后入库 |
| 人脸/活体 | 阿里云实人认证（可选） | 配置 ALIYUN_ACCESS_KEY_* 后启用，否则 mock |
| 测试 | Node test + supertest | 后端 API 自动化测试（26 条）；Playwright E2E（web/e2e） |

## 五、关键流程图

### 5.1 管理端登录流程

```mermaid
sequenceDiagram
  participant U as 用户
  participant P as 前端页面
  participant A as 后端 /api/auth
  participant D as 数据库

  U->>P: 打开 /login 或任意 /pc/* 页
  P->>P: 检查 localStorage labor_token
  alt 无 token
    P->>U: 跳转 /login
    U->>P: 输入账号密码
    P->>A: POST /api/auth/login
    A->>D: 校验 user 表
    D-->>A: 用户信息
    A-->>P: token + user
    P->>P: 存 token，跳转 /pc/dashboard
  else 有 token
    P->>A: 请求业务接口带 Bearer token
    A->>A: 校验 JWT
    A-->>P: 返回数据
  end
```

### 5.2 工人端登录流程

```mermaid
sequenceDiagram
  participant W as 工人
  participant P as 前端 H5
  participant A as 后端 /api/auth
  participant D as 数据库

  W->>P: 打开 /h5（无工人 token 则跳 /h5/login）
  P->>P: 检查 labor_worker_token
  alt 无 token
    P->>W: 显示工号+姓名登录
    W->>P: 输入工号、姓名
    P->>A: POST /api/auth/worker-login
    A->>D: 按工号或姓名查 person
    alt 找到人员
      A-->>P: token + person
      P->>P: 存 token，跳转 /h5 首页
    else 未找到
      A-->>P: 401 未找到
    end
  end
```

### 5.3 考勤 → 结算主流程

```mermaid
flowchart LR
  A[Excel 考勤表] -->|上传| B[考勤数据接入]
  B -->|写入| C[(attendance 表)]
  C -->|按周期汇总| D[生成结算单]
  D -->|写入| E[(settlement 表)]
  E --> F[管理端确认/驳回]
  E --> G[工人端待确认列表]
  G --> H[工人确认或驳回]
```

- 管理端在「考勤数据接入」上传 Excel，系统解析后写入 `attendance`。
- 在「结算单确认」页选择周期，点击生成，系统按周期从考勤汇总生成 `settlement` 待确认记录。
- 管理端可在此页做确认/驳回；工人端在「我的薪资」看到待确认项并确认或驳回。

### 5.4 合同发起到签署流程

```mermaid
flowchart LR
  A[管理端合同发起] -->|选人+标题+截止日| B[(contract_instance)]
  B -->|状态: 待签署| C[工人端待签列表]
  C --> D[工人点击签署]
  D --> E[POST /contract/sign/:id]
  E --> F[状态: 已签署]
  F --> G[person.contract_signed=1]
  F --> H[管理端合同归档可见]
```

- 管理端在「合同发起」多选人员、填标题与截止日，为每人生成一条合同实例（待签署）。
- 工人在 H5「我的合同」看到待签列表，点「去签署」进入签署页，确认后调用签署接口，状态变为已签署，并回写人员签约标识；管理端「合同归档」可查已签署记录。
