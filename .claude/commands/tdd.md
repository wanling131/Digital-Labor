# TDD (Test-Driven Development) 工作流

严格遵循红-绿-重构循环开发业务代码。

## 工作流程

### 第一阶段：Red（红灯）
1. **先写测试**：编写一个失败的测试用例
2. **运行测试**：确认测试失败（红灯）
3. **记录预期**：明确测试应该验证什么行为

### 第二阶段：Green（绿灯）
1. **写最少代码**：只写足以让测试通过的代码
2. **运行测试**：确认测试通过（绿灯）
3. **不过度设计**：不要在此阶段添加额外功能

### 第三阶段：Refactor（重构）
1. **检查代码**：寻找重复、冗余、可优化的地方
2. **重构代码**：在测试保护下安全重构
3. **运行测试**：确保重构后测试仍通过（保持绿灯）

## 执行检查清单

- [ ] 测试文件已创建
- [ ] 测试初始运行失败（红灯确认）
- [ ] 实现代码已编写
- [ ] 测试运行通过（绿灯确认）
- [ ] 代码已重构（如需要）
- [ ] 最终测试仍通过

## 禁止行为

- 不写测试直接写实现代码
- 测试未失败就写实现代码
- 重构后不运行测试验证

## 项目测试命令

```bash
# 后端测试
cd server && python -m pytest tests/ -v

# 前端测试
cd web && npm run test
```
