# 人脸/活体配置说明

> 本系统人脸核验支持**阿里云实人认证**。配置 AccessKey 后自动启用真实接口，未配置时使用本地模拟（便于开发与演示）。文档索引见 [docs/README.md](./README.md)。
>
> **说明**：阿里云实人认证需完成**企业认证**才能开通。未完成前可先使用内置 mock（人脸验证会直接返回通过），待企业认证通过后再按本文配置真实接入。

---

## 一、配置方式概览

| 方式 | 说明 |
|------|------|
| **已配置密钥** | 在 server 目录配置 `ALIYUN_ACCESS_KEY_ID`、`ALIYUN_ACCESS_KEY_SECRET` 后，人脸/活体请求将走阿里云接口 |
| **未配置或关闭** | 使用内置 mock，始终返回“通过”，便于联调与演示 |
| **显式关闭** | 设置 `FACE_VERIFY_ENABLED=false` 可强制使用 mock（即使已配置密钥） |

## 二、阿里云侧准备

### 2.1 开通产品

1. 登录 [阿里云控制台](https://www.aliyun.com/)
2. 打开 **实人认证**（原 云盾实人认证 / ID Verification）
   - 文档：<https://help.aliyun.com/product/28416.html>
3. **开通前需完成阿里云企业认证**；开通后按需选择 **金融级实人认证** 或 **活体检测** 等能力（以控制台实际产品为准）

### 2.2 创建 RAM 子账号并授权

1. 进入 **访问控制 RAM** → **用户** → **创建用户**
2. 勾选 **OpenAPI 调用**，保存并记录 **AccessKey ID**、**AccessKey Secret**
3. 为该用户授权策略：**AliyunCloudAuthFullAccess**（实人认证完整权限），或按最小权限仅开放 `cloudauth:InitFaceVerify`、`cloudauth:DescribeFaceVerify` 等

### 2.3 获取密钥

- **AccessKey ID**：形如 `LTAI5t***`（约 24 位字母数字），在 RAM 用户创建时勾选「OpenAPI 调用」后生成
- **AccessKey Secret**：与 AccessKey ID 成对出现，创建时仅展示一次，请妥善保存

### 2.4 和「用户名/密码」的区别

本系统**只认阿里云 RAM 的 AccessKey 对**，不认控制台登录用的「用户名/密码」，也不认其他系统给的「userName + password」或「API Key」。

| 您可能拿到的 | 是否用于本系统 |
|--------------|----------------|
| 阿里云控制台登录账号、密码 | 否，仅用于登录控制台 |
| 其他系统提供的 `userName: digital-labor-app`、`password: AP:xxx` | 否，格式与阿里云 OpenAPI 不兼容 |
| RAM 用户 → 创建 AccessKey 得到的 **AccessKey ID** + **AccessKey Secret** | 是，填到下方 `.env` 中 |

若您只有「userName / password」形式的凭证，需要到 **阿里云控制台 → 访问控制 RAM → 用户** 中为相应 RAM 用户**创建 AccessKey**，才会得到 `ALIYUN_ACCESS_KEY_ID` 和 `ALIYUN_ACCESS_KEY_SECRET`。实人认证等 OpenAPI 调用必须使用这对密钥做签名，无法用用户名+密码替代。

## 三、本系统环境变量配置

### 3.1 后端（server）

在 **server** 目录下创建或编辑 `.env` 文件（不要提交到 Git）：

```bash
# 阿里云人脸/活体（实人认证）
ALIYUN_ACCESS_KEY_ID=你的AccessKeyId
ALIYUN_ACCESS_KEY_SECRET=你的AccessKeySecret

# 可选：是否启用人脸核验，不填或 true 表示启用；false 表示强制使用 mock
FACE_VERIFY_ENABLED=true
```

### 3.2 配置生效方式

- 使用 `npm run dev` / `node server/index.js` 等方式启动时，会自动加载 server 目录下的 `.env`
- 若使用 pm2、Docker 等，请在对应环境中配置上述变量

## 四、验证是否生效

### 4.1 看启动日志

启动后端服务后查看控制台：

- **使用阿里云**：会看到 `[FaceVerify] 使用阿里云人脸验证服务`
- **使用模拟**：会看到 `[FaceVerify] 使用模拟人脸验证服务（开发测试用）`

### 4.2 调接口查询

请求（需带管理端或工人端 Token）：

```http
GET /api/sys/feature-status
```

响应示例：

```json
{
  "faceVerify": "aliyun"
}
```

或（未配置密钥时）：

```json
{
  "faceVerify": "mock"
}
```

前端可根据 `faceVerify === 'aliyun'` 显示“已接入真人脸”等标识。

### 4.3 实际核验流程

- **H5 激活 / 个人中心 / 结算确认**：进入人脸验证 → 允许摄像头 → 拍照后由前端将 base64 提交到 `POST /api/person/face-verify`
- 配置了密钥时，该请求会转发至阿里云；未配置时走 mock，直接返回通过

## 五、接口说明（当前实现）

| 模式 | 说明 | 请求体要点 |
|------|------|------------|
| **living** | 活体检测（当前为单张照片检测） | `mode: "living"`, `image`: 拍照 base64, 可选 `person_id` |
| **compare** | 1:1 人脸比对 | `mode: "compare"`, `image`, `target_image`, 可选 `person_id` |
| **full** | 实人认证（姓名+身份证+设备 MetaInfo） | `mode: "full"`, `cert_name`, `cert_no`, `meta_info`, 可选 `person_id` |

- 通过后若传了 `person_id`，会更新人员表 `face_verified = 1`、`face_verified_at`
- 限流：见 `server/middleware/rateLimit.js` 中 `faceVerify` 配置

## 六、阿里云 API 与代码对应关系

当前 `server/lib/faceVerify.js` 中使用的逻辑为：

- 未配置密钥时：走 **mock**，不请求阿里云
- 已配置密钥时：调用阿里云 **cloudauth** 相关接口（具体 Action 以代码为准）

若阿里云控制台开通的是 **实人认证（InitFaceVerify / DescribeFaceVerify）**，且接口版本、参数与当前代码不一致，需要：

1. 查阅 [阿里云实人认证 API 文档](https://help.aliyun.com/document_detail/60410.html)
2. 在 `server/lib/faceVerify.js` 中调整 **Action 名称、请求参数、Endpoint** 以与文档一致

## 七、常见问题

| 现象 | 可能原因 | 处理 |
|------|----------|------|
| 一直走 mock | 未配置或未加载 .env | 确认 .env 在 server 目录、变量名正确、重启服务 |
| 控制台报错“阿里云API调用失败” | Key 错误、未授权、网络或接口参数不符 | 检查 Key、RAM 授权、阿里云文档与代码是否一致 |
| 前端提示“验证失败” | 网络异常或阿里云返回不通过 | 看后端日志中阿里云返回的 Message/Reason |
| 希望临时关闭真人脸 | 生产需临时切回演示 | 设置 `FACE_VERIFY_ENABLED=false` 并重启 |

---

## 八、安全建议

1. **AccessKey** 仅在后端使用，不要写入前端或公开仓库
2. 使用 **RAM 子账号** + 最小权限，不要使用主账号 Key
3. 生产环境建议将 `.env` 设为 600 权限，并通过密钥管理服务或 CI 注入变量
