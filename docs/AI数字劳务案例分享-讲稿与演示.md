# 一、怎么用 AI 提效（做这个项目的体会）

做数字劳务这个项目时，用 **AI 工具**（如 Cursor、Copilot 等）写代码、写文档、查错、整理讲稿，确实能少写不少样板代码、减少反复改稿。下面几条是真实用过、能落地的做法，大家可以直接照着试。

---

## 1. 用 AI 做什么：写代码、写文档、查错、整理材料

**说**：  
**写代码**：新组件（如本项目里的 `web/components/h5/face-verify.tsx` 人脸核验）、新页面（如人员档案列表、考勤报表）、接口调用和 Excel 解析等，先让 AI 按「当前项目技术栈、目录、命名」生成一版，再自己改几处对接现有 api()、路由，比从零手写快。重复多的——同风格的列表页、表单页、接口封装——特别适合让 AI 打底。**写文档**：本项目 `docs/` 下有部署说明、环境变量示例、API 说明等，可以先把代码或路由给 AI，让它生成初稿，再自己按「实际命令、实际接口」跑一遍，把不对的地方改掉，就能用。**查错**：把报错信息、堆栈、相关代码片段贴给 AI，让它分析可能原因、给修改建议，依赖问题、环境差异、异步报错这类经常能快速缩小范围。**整理材料**：像这份讲稿，先列「讲什么、分几块、每块要点」，让 AI 搭框架和段落，再按自己项目改具体例子和话术，能省时间。

**本项目里真实存在的可对照**：`web/components/h5/face-verify.tsx`（摄像头 + base64 调 POST /api/person/face-verify）、`web/app/pc/personnel/archive/page.tsx`（人员列表与筛选）、`web/app/pc/attendance/` 下考勤导入与报表、以及当前这份讲稿文档，都是「AI 出初版 + 人按项目改」可以复现的用法。

**体会**：有规律、能说清楚的任务交给 AI 打底；业务规则复杂、强依赖领域判断的部分人主导，AI 只辅助细节。

**演示**：无。

## 2. 提效的关键：说清楚要什么、给足上下文

**说**：  
AI 要能帮上忙，**需求要说清楚**。本项目里管理端登录用 **api()**、token 存 **labor_token**（见 `web/lib/api.ts`），成功跳 **/pc/dashboard**；如果让 AI 写登录页，就要说清「Next.js、用现有 api()、token 键 labor_token、失败提示错误、成功跳 /pc/dashboard」，而不是只说「写个登录页」。建议带上：**谁用**（管理端 / 工人端）、**现有约定**（请求用 api() 还是 apiWorker()、token 键名）、**输入输出和边界**（成功失败怎么表现、网络失败要不要提示）。**上下文要给足**：在 Cursor 里 **@ 相关文件**（如 @ web/lib/api.ts、@ 某个已有列表页），或贴一段现有组件、接口调用的写法，说「按这个风格来」，生成的代码才容易直接嵌进项目，少出现「语法对但接不上、命名不统一」。

**反面例子**：如果只说「加个考勤导入」，AI 可能写出一套和现有接口不一致的实现。本项目里考勤导入实际是 **POST /api/attendance/import**，form-data 传 file，返回 **{ ok, count }**；先把这些约定写进提示里，再让 AI 写前端上传和解析，更容易一次对上。

**可照着做的一句提示示例**（要加一个和人员列表同风格的考勤报表页时）：  
「@ web/app/pc/personnel/archive/page.tsx，按这个页面的表格和 api() 调用方式，写一个考勤报表页，路径放在 web/app/pc/attendance/report/，接口是 GET /api/attendance/report，query 用 page、pageSize、start、end，返回 { list, total }。」  
这样 AI 会参照现有列表页的写法，接口和路径也是项目里真实存在的，改起来有据可查。

**体会**：多花半分钟把「要什么、现有约束是什么」说清楚，能省后面很多返工；**会描述需求，本身就是提效**。

**演示**：无。

## 3. 小步迭代，别一次让 AI 干一大块

**说**：  
不要一次让 AI「把整个考勤模块写完」或「把全部接口文档生成完」。**拆成小步**：比如「先加这个接口」「先改这个组件的这一块」「先写这一节的文档」「先把这一页的表格和筛选做出来」，每一步都有明确输入和可验证的输出，改完跑一下、看效果，再继续下一步。这样 AI 不容易「跑偏」——大段生成时它容易漏掉你项目里已有的约定或依赖，小步的话你每步都能纠正方向；而且小步改动用 Git 也好回滚，哪一步出问题一目了然。**本项目里的真实拆分方式**：合同发起在 `web/app/pc/contract/initiate/`，可以先让 AI 做「多选人员 + 截止日表单 + 调人员列表接口」，跑通后再加「发起接口 POST /api/contract/launch 和成功跳转」；写 docs 时按「先 Nginx 配置这一段」「再环境变量这一节」一节节来，每节对一下实际工程再继续，避免整篇和实际对不上。

**体会**：AI 提效的前提是「每一步都可控、可验」；大块交给 AI 一次搞定，看起来省事，往往后面要花更多时间修和补漏。

**演示**：无。

## 4. 结果必验：跑一遍、测一遍再算完成

**说**：  
AI 生成的代码或文档**不能直接当最终版**，必须自己验一遍。**代码**：`npm run dev` 能起、登录能进、相关页面能点；调接口的用浏览器 Network 或实际点一下看请求是否带 token（本项目是 Authorization: Bearer）、返回是否和 backend 约定一致（如 list、total、分页参数 page/pageSize）。真实容易出的问题包括：AI 用了 fetch 没走项目里的 api()、所以没带 labor_token；或分页参数名和 backend 不一致（如 backend 要 pageSize 却写成了 limit）；这些只有跑一遍、点一遍才能发现。**文档**：部署说明里的命令在本地或目标环境跑一遍，环境变量和 docs/环境变量示例.md 对一下，漏掉的必填项补上；AI 容易写成「通用版」，把你项目特有的步骤或路径写丢。

**谁对结果负责**：用 AI 提效，**质量门在自己手上**；对外交付的代码和文档，最终负责的是人。

**体会**：不验就上容易埋坑；验一遍成本不高，能避免后面修 bug、改文档。

**演示**：无。

## 5. 哪些场景 AI 特别能提效、哪些要人主导

**说**：  
**AI 特别能提效的场景**：重复结构多的——比如多个列表页、表单页、类似风格的接口封装；有现成规律可描述的——比如「和人员列表一样，只是接口换成考勤、字段换成日期和工时」；查错和排查——贴报错、堆栈、配置，让 AI 给可能原因和修改建议；文档和讲稿——有代码或要点就能生成初稿，再按实际改。**更适合人主导、AI 辅助的场景**：业务规则特别复杂、强依赖领域知识或公司内部约定的（比如结算规则、审批流程）；和安全性、权限、数据一致性直接相关的逻辑，人设计、AI 帮忙实现细节；以及最终要不要采纳 AI 的建议、怎么改，还是人拍板。

**一句话**：有规律、能说清楚的交给 AI 打底，**关键判断和最终质量还是人把关**。

**演示**：无。

## 6. 用什么工具、怎么配合

**说**：  
工具按习惯选、能组合用。**Cursor**：和代码同环境，能 **@ 文件、@ 目录**，边看边问「这段怎么改」「按这个风格加一个类似页面」，适合改现有工程、对齐本项目这种已有 api()/apiWorker()、已有目录结构的代码库。**Copilot**：在编辑器里按注释或函数名补全，适合重复结构（如又一个表格列、又一个 api() 调用）。**ChatGPT 等**：适合整理长文本、搭文档或讲稿框架、把报错贴进去问可能原因。本项目就是在 Cursor 里写代码、改代码，文档和讲稿可以先用 AI 搭框架，再在 Cursor 里对着 `docs/`、`web/app/` 真实路径和接口改一版，保证和项目一致。

**体会**：工具是手段，**说清楚需求、给足上下文、小步迭代、结果必验**这几条，用哪个工具都适用。

**演示**：无。

## 7. 小结：用 AI 提效的几条

**说**：  
总结一下**怎么用 AI 提效**：  

- **用在哪儿**：写代码、写文档、查错、整理讲稿等有规律、可描述清楚的任务；重复结构多、有现成约定可参照的，AI 特别能提效。  
- **怎么说**：说清楚要什么（谁用、技术栈、输入输出、边界），给足项目和技术栈上下文（@ 文件、贴现有写法），AI 产出才贴项目。  
- **怎么控**：小步迭代、每一步可验证，别一次让 AI 干一大块；关键判断和最终质量人把关。  
- **怎么收口**：结果必验——代码跑一遍、流程点一遍，文档对一遍实际行为，质量门在自己手上。

这样用 AI 是**提效**而不是「替人」：少写样板代码、少改几轮稿，质量靠人验一遍兜底。本项目里写组件、对接口、写 docs、准备这场分享，都是按「说清楚、给上下文、小步改、结果必验」来做的，大家可以直接拿这几条在自己项目里试，有更好用法也欢迎会后交流。

**演示**：无。

## 2.2 管理端实现的功能

**说**：  
管理端我们实现了这些：


| 模块       | 实现内容                                                 |
| -------- | ---------------------------------------------------- |
| **人员档案** | 人员增删改查、按状态/组织/在岗筛选、分页；状态流转（预注册→已实名→已签约→已进场→已离场）、批量变更 |
| **认证管理** | 按人脸采集、核验状态筛选人员，谁已实名、谁未完成一目了然；审核通过/驳回                 |
| **合同**   | 合同模板、按人多选发起、截止日；签约状态列表；归档检索、作废、下载；工人端待签/已签与签署        |
| **考勤**   | Excel 导入考勤、自动排重；工时报表、按条件查询与导出                        |

| **现场管理** | 离场登记、在岗看板按组织汇总在岗人数                                   |
| **数据看板** | KPI 卡片、近 7/30/90 天趋势（新增人员、签约数、考勤人次、工时）               |
| **系统管理** | 组织树、用户管理、批量导入、操作日志等                                  |
