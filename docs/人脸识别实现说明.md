# 人脸识别实现说明

本文说明本项目中**人脸相关**的现状，以及若要接入**真人脸识别/活体检测**可以怎么做、需要下载什么、如何实现。

---

## 一、当前项目里的“人脸”现状

- **后端**：`POST /api/person/face-verify` 为**占位接口**，只做参数校验，不调用任何算法或第三方，直接返回 `{ ok: true }`。
- **前端**：H5 个人中心「人脸实名认证」、合同签署前、待确认结算单「确认」前会调用该接口，走一遍占位流程。
- **目的**：业务逻辑和接口形态已预留，后续只需在 `face-verify` 内替换为真实的人脸/活体服务即可。

**不需要**在本地下载人脸模型或 SDK，当前就能跑通整条流程（占位通过）。

---

## 二、真人脸识别一般怎么实现（两种方式）

### 方式一：云 API（推荐，无需下载模型）

**特点**：不在本机跑算法，不下载模型，只调厂商提供的 HTTP 接口。

| 项目     | 说明 |
|----------|------|
| 是否需要下载 | **不需要**下载模型或 SDK，只需在项目里用 `axios`/`fetch` 调接口。 |
| 典型能力 | 活体检测（眨眼/张嘴/转头等）、人脸比对（1:1）、人脸搜索（1:N）。 |
| 常见厂商 | 阿里云实人认证、腾讯云人脸核身、百度智能云人脸识别、华为云等。 |
| 使用方式 | 在对应云平台开通服务 → 获取 API Key/Secret → 后端用 Key 调 REST API，传图片（base64 或 URL）。 |

**实现思路**：

1. 在阿里云/腾讯云/百度等开通「人脸核身」或「实人认证」类产品。
2. 获取 **AccessKey / Secret** 或 **API Key / Secret Key**（按厂商文档）。
3. 在**后端**（如 `server/routes/person.js` 的 `face-verify` 中）：
   - 接收前端上传的**人脸图（base64）**或**活体检测结果 token**（视厂商流程而定）；
   - 用 Node 的 `https` 或 `axios` 请求厂商接口（活体校验 + 可选 1:1 与身份证照/底库照比对）；
   - 根据返回结果决定是否返回 `{ ok: true }` 或失败信息。
4. **前端**：  
   - 要么用厂商提供的 **H5/Web SDK**（通常是一个 js，按文档引入），在页面里调起摄像头、做活体动作、拿到图片或 token 再发给你们后端；  
   - 要么使用厂商的** H5 拉活体页面**（跳转或 iframe），完成后回调你们后端。

**总结**：不下载“某个人脸模型文件”，只需要在后台写调用云 API 的代码，前端按厂商文档接入他们的 H5/Web 能力即可。

---

### 方式二：本地 SDK / 离线模型

**特点**：在你们服务器或终端上跑算法，需要集成 SDK 或模型文件。

| 项目     | 说明 |
|----------|------|
| 是否需要下载 | **需要**：下载并部署厂商的 **SDK** 或 **离线模型**（如 .so/.dll + 模型文件）。 |
| 典型能力 | 活体检测、人脸检测、1:1 比对等，全部在本地执行。 |
| 常见厂商 | 虹软 ArcFace、腾讯云离线 SDK、百度离线等。 |
| 使用方式 | 下载对应平台（Linux/Windows/Android/iOS）的 SDK，按文档在 Node 里用 addon 调用，或在 C++/Java 服务里调用再对 Node 提供接口。 |

**实现思路**：

1. 从厂商官网下载对应系统的 **SDK 包**（通常包含动态库 + 头文件 + 示例）。
2. 可能还需要下载/申请 **授权文件**（.lic 等），用于激活 SDK。
3. 在**后端**用 C++ 或厂商提供的 Node 绑定封装一个服务，接收图片 → 调 SDK（活体 + 比对）→ 返回是否通过。
4. 你们现有的 `POST /api/person/face-verify` 再改为调用这个本地服务，而不是直接返回成功。

**总结**：需要“下载”的是**厂商的 SDK 安装包 + 授权文件**，不是某个通用开源模型；部署和运维成本比云 API 高，一般适合内网或对隐私/离线要求高的场景。

---

## 三、本项目中推荐的做法（云 API）

劳务场景下，**活体检测 + 人脸比对**用云 API 即可满足，且无需在你们机器上下载或部署模型：

1. **选一家云厂商**（如阿里云实人认证、腾讯云人脸核身），在控制台开通服务。
2. **不需要**在项目里“下载某个人脸模型”；只需要：
   - 在**后端**安装 HTTP 客户端（如 `axios`），并配置好 API Key/Secret（建议放 `.env`，如 `FACE_VERIFY_API_KEY`）。
   - 在 **server/routes/person.js** 的 `face-verify` 中：
     - 接收前端传来的**人脸图 base64**（或厂商要求的活体 token）；
     - 调云厂商接口做**活体检测**（必要时再做人脸 1:1 比对）；
     - 根据云 API 返回结果返回 `{ ok: true }` 或 `{ ok: false, message: '...' }`。
3. **前端**：
   - 使用厂商提供的 **H5/Web 活体 SDK**（按文档引入一个 js），在页面里调起摄像头、完成动作、拿到人脸图或 token；
   - 将结果通过现有接口发给后端（例如 `POST /api/person/face-verify`，body 里带 `image` 或厂商返回的 token）。

这样，**“需要下载什么”**的答案就是：  
- **不下载**人脸模型；  
- 若用云 API：只需在后台写调用代码，前端按厂商文档引入他们的 **H5/Web SDK**（通常是一个脚本或 npm 包）；  
- 若用本地 SDK：需要下载**厂商提供的 SDK 安装包和授权文件**，并在服务器上部署。

## 四、对接时后端要改哪里

当前占位逻辑在 **server/routes/person.js** 的 `POST /face-verify`：

```js
// 当前：占位，直接通过
router.post('/face-verify', (req, res) => {
  const { image, person_id } = req.body || {}
  if (!image && !person_id) {
    return res.status(400).json({ message: '请提供 image（base64）或 person_id' })
  }
  res.json({ ok: true, message: '核验通过（占位）' })
})
```

对接真人脸时建议：

1. 从环境变量读取配置（如 `FACE_VERIFY_ENABLED`、`FACE_VERIFY_API_KEY`、厂商 API 地址）。
2. 若 `FACE_VERIFY_ENABLED !== 'true'`，可保持现有占位逻辑（直接返回通过）。
3. 若启用真人脸：
   - 用 `image`（base64）或厂商要求的参数调云 API（活体 + 可选 1:1 比对 `person_id` 对应的人员底库照）；
   - 根据云 API 返回判断是否通过，再返回 `{ ok: true }` 或 4xx + message。

环境变量示例已在 **docs/环境变量示例.md** 中预留（如 `FACE_VERIFY_API_KEY`），可直接使用。

## 五、简要对比

| 项目           | 云 API 方式        | 本地 SDK/模型方式   |
|----------------|--------------------|----------------------|
| 是否要下载模型 | 不需要             | 需要（厂商 SDK+授权）|
| 后端改动       | 调 HTTP 接口       | 集成 SDK/本地服务   |
| 前端           | 用厂商 H5/Web SDK  | 同左或自采图给后端   |
| 成本           | 按调用量计费      | 授权费 + 服务器资源 |
| 隐私/离线      | 数据走公网         | 可完全内网/离线     |

若你确定选用某一家云厂商（如阿里云或腾讯云），可以说一下厂商名称，我可以按该厂商的接口文档帮你把 `face-verify` 的请求/响应格式和错误处理写成一版示例代码（仍不包含真实 Key）。
