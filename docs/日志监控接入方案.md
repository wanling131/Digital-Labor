# Digital Labor · 日志监控接入方案

本文档给出「日志监控」的接入方案：推荐工具、分层设计、接入步骤及环境变量，便于按阶段落地。现有**操作日志（op_log）**为审计用途，本方案在此基础上补充**错误追踪、运行监控与告警**。

---

## 一、现状与目标

| 已有 | 缺失 |
|------|------|
| 操作日志表 `op_log`：记录谁、在何模块、做了何操作、成功/失败；PC 端可查、可筛选、可导出 | 前端/后端异常自动上报与告警 |
| 后端健康检查 `GET /api/health` | 接口错误率、延迟等指标监控与可视化 |
| 无集中日志检索 | 错误率/健康检查失败等触发钉钉/邮件/短信告警 |

**目标**：在不影响现有 op_log 的前提下，增加（1）错误监控、（2）简单运行监控与告警，可选（3）集中日志。

---

## 二、推荐工具与分层

| 层级 | 用途 | 推荐工具 | 优先级 |
|------|------|----------|:------:|
| **L1 错误监控** | 前端/后端异常自动捕获、堆栈、上下文、告警 | **Sentry**（或 阿里云 ARMS / 腾讯云前端监控） | 高 |
| **L2 运行与告警** | 健康检查、接口可用性、简单指标与告警 | **自建轮询 + 钉钉/邮件** 或 **Uptime Robot / 云监控** | 中 |
| **L3 集中日志（可选）** | 多机/多容器日志统一检索、大盘 | **Loki + Grafana** 或 **ELK** 或云原生日志 | 低 |

建议实施顺序：**先 L1（Sentry）→ 再 L2（健康检查 + 告警）→ 有需要再做 L3**。

---

## 三、方案一：Sentry 错误监控（推荐先做）

### 3.1 适用与收益

- 前端（Next.js）：未捕获异常、Promise rejection、部分 4xx/5xx 请求可上报。
- 后端（Node/Express）：未捕获异常、主动 `captureException`。
- 支持：错误分组、堆栈、用户/请求上下文、环境标签、邮件/钉钉等告警。

### 3.2 注册与创建项目

1. 打开 [sentry.io](https://sentry.io)，注册并创建 Organization。
2. 创建两个 Project（类型选对）：
   - **labor-web**：平台选 **Next.js**（或 JavaScript）。
   - **labor-api**：平台选 **Node.js / Express**。
3. 在项目设置中拿到 **DSN**（形如 `https://xxx@xxx.ingest.sentry.io/xxx`），后续配置用。

### 3.3 后端（Express）接入步骤

1. **安装依赖**（在 `server` 目录下）：

```bash
cd server
npm install @sentry/node
```

2. **在入口最前面初始化**（在 `server/app.js` 最顶部，任何 require 其他路由之前）：

```javascript
import * as Sentry from '@sentry/node'

Sentry.init({
  dsn: process.env.SENTRY_DSN,
  environment: process.env.NODE_ENV || 'development',
  tracesSampleRate: 0.2,
  enabled: !!process.env.SENTRY_DSN,
})
```

3. **错误处理中间件**（在现有全局错误处理之后挂一个 Sentry 上报）：

在 `app.js` 中注册完所有路由后，增加：

```javascript
app.use(Sentry.Handlers.errorHandler())
```

若已有 `app.use((err, req, res, next) => { ... })`，可在其内部调用 `Sentry.captureException(err)`，再 `next(err)` 或返回 500。

4. **请求上下文**（可选）：在鉴权中间件里设置用户，便于 Sentry 里按用户过滤：

```javascript
if (req.user) {
  Sentry.setUser({ id: String(req.user.userId), username: req.user.username })
}
```

5. **环境变量**（`server/.env` 或根目录）：

```bash
SENTRY_DSN=https://xxx@xxx.ingest.sentry.io/xxx
```

不设或为空则不上报，便于本地开发不污染 Sentry。

### 3.4 前端（Next.js）接入步骤

1. **安装依赖**（在 `web` 目录下）：

```bash
cd web
npm install @sentry/nextjs
```

2. **执行 Sentry 向导（推荐）**：

```bash
npx @sentry/wizard@latest -i nextjs
```

按提示选择项目、填入 DSN、自动生成 `sentry.client.config.ts`、`sentry.server.config.ts`、`sentry.edge.config.ts` 及 `next.config.js` 的修改。未用向导则需手动在 Next 配置里注入 Sentry 的 webpack 与运行时配置。

3. **环境变量**（`web/.env.local`）：

```bash
NEXT_PUBLIC_SENTRY_DSN=https://xxx@xxx.ingest.sentry.io/xxx
SENTRY_DSN=https://xxx@xxx.ingest.sentry.io/xxx
```

4. **可选**：在关键接口请求失败时主动上报（例如在 `web/lib/api.ts` 的 catch 里）：

```typescript
if (process.env.NODE_PUBLIC_SENTRY_DSN && err instanceof Error) {
  const Sentry = await import('@sentry/nextjs')
  Sentry.captureException(err, { extra: { path, status: res?.status } })
}
```

### 3.5 告警配置（Sentry 控制台）

- 在 Sentry 项目设置中配置 **Alerts**：新建 Rule，例如「某项目错误数 1 小时内 > N 则告警」。
- 集成 **Slack / 钉钉 / 邮件**：在 Integration 里添加，并在 Alert 中选择对应渠道。

## 四、方案二：运行监控与告警（健康检查 + 钉钉/邮件）

### 4.1 思路

不依赖复杂 APM，仅对「服务是否存活」「接口是否正常」做轮询，失败时触发告警。

### 4.2 方式 A：第三方监控（零代码）

- 使用 **Uptime Robot**、**阿里云云监控**、**腾讯云拨测** 等。
- 配置 HTTP(s) 监控：`https://你的域名/api/health`，间隔 5 分钟。
- 失败时通过其内置邮件/短信/Webhook 告警；Webhook 可接到钉钉机器人。

### 4.3 方式 B：自建简单轮询脚本 + 钉钉

1. **钉钉群**：建群 → 添加「自定义机器人」→ 选「加签」→ 得到 `Webhook URL` 和 `SECRET`。

2. **脚本示例**（可放在 `scripts/health-check.mjs`，由 crontab 或 CI 定时跑）：

```javascript
const HEALTH_URL = process.env.HEALTH_URL || 'http://localhost:3000/api/health'
const DING_WEBHOOK = process.env.DING_WEBHOOK
const DING_SECRET = process.env.DING_SECRET

async function check() {
  const res = await fetch(HEALTH_URL).catch(() => null)
  const ok = res?.ok && (await res.json())?.ok
  if (ok) return
  if (!DING_WEBHOOK) return console.error('Health check failed')
  const timestamp = Date.now()
  const sign = require('crypto').createHmac('sha256', DING_SECRET).update(`${timestamp}\n${DING_SECRET}`).digest('base64')
  await fetch(DING_WEBHOOK, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      msgtype: 'text',
      text: { content: `[劳务系统] 健康检查失败: ${HEALTH_URL}` },
      sign,
      timestamp,
    }),
  })
}
check()
```

3. **环境变量**：`HEALTH_URL`、`DING_WEBHOOK`、`DING_SECRET`（生产环境填真实域名与机器人信息）。

4. **定时**：例如服务器上 `crontab -e` 每 5 分钟执行一次 `node scripts/health-check.mjs`。

### 4.4 方式 C：后端在启动/关键错误时主动推钉钉（可选）

在 Express 全局错误处理里，当 `NODE_ENV=production` 且发生 5xx 时，调用钉钉 Webhook 发送一条「接口异常：path + message」。需注意频率限制与去重，避免刷屏。

## 五、方案三：集中日志（可选，后期）

当有多实例、多容器或需要按日志全文检索时再考虑。

| 方案 | 特点 | 适用 |
|------|------|------|
| **Grafana Loki** | 轻量、与 Grafana 集成好、按标签检索 | 自建、K8s/ Docker |
| **ELK（Elasticsearch + Logstash + Kibana）** | 功能全、检索强 | 日志量大、需复杂检索 |
| **云原生日志** | 阿里 SLS、腾讯 CLS、华为 LTS | 上云且希望免运维 |

**步骤概要**：

1. 应用侧：使用 **pino** / **winston** 等打 JSON 日志到 stdout，或写文件。
2. 采集：Filebeat / Promtail / 云日志 Agent 采集文件或 stdout，发送到 Loki/ES。
3. 查询与告警：在 Grafana/Kibana 做大盘与告警规则，对接钉钉/邮件。

当前单机、单实例可暂不实施，以 Sentry + 健康检查告警为主即可。

## 六、环境变量汇总（与现有 .env 合并）

在 **`docs/环境变量示例.md`** 或实际 `.env` 中可增加：

```bash
# ---------- 日志监控（可选） ----------
# Sentry 后端
SENTRY_DSN=

# Sentry 前端（web 项目 .env.local）
# NEXT_PUBLIC_SENTRY_DSN=
# SENTRY_DSN=

# 健康检查告警（自建脚本用）
# HEALTH_URL=https://your-api-domain.com/api/health
# DING_WEBHOOK=
# DING_SECRET=
```

未配置则不启用对应能力，不影响现有功能。

## 七、与现有操作日志（op_log）的关系

| 能力 | op_log | 本方案（Sentry + 健康告警） |
|------|--------|-----------------------------|
| 记录内容 | 谁、模块、动作、结果、请求体摘要 | 异常堆栈、请求上下文、环境 |
| 用途 | 审计、合规、事后排查「谁做了什么」 | 故障发现、错误定位、可用性告警 |
| 存储 | SQLite 表，PC 端查询/导出 | Sentry 云端 / 钉钉消息 |
| 触发方式 | 每次请求由中间件写入 | 异常发生 / 健康检查失败时上报或推送 |

两者互补：**op_log 保留不动**，本方案只做「错误与可用性监控」，不替代业务操作日志。

---

## 八、实施顺序建议

1. **第一阶段**：接入 Sentry（后端 + 前端），配置 DSN 与告警渠道；本地不配 DSN，生产再开。
2. **第二阶段**：配置健康检查告警（Uptime Robot 或自建脚本 + 钉钉），确保接口不可用时能收到通知。
3. **第三阶段**（按需）：多实例或日志检索需求上来后，再引入 Loki/ELK 或云日志。

文档版本：与当前技术栈（Node/Express + Next.js）一致；具体 DSN、Webhook 等以实际创建为准。
