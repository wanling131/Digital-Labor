# Digital Labor · 系统嵌入与对接设计

本文档描述 P2 系统嵌入运管平台、模块化集成及与其他系统对接的设计要点。

---

## 一、系统嵌入运管平台

### 1.1 独立路由前缀

- **前端**：通过 `NEXT_PUBLIC_LABOR_BASE_PATH` 配置路由前缀（如 `/labor`），页面变为 `/labor/pc/dashboard`、`/labor/h5` 等。
- **后端**：通过 `LABOR_API_PREFIX` 配置 API 前缀（如 `/labor`），接口变为 `/labor/api/auth`、`/labor/api/person` 等。

### 1.2 环境变量注入

| 变量 | 说明 | 示例 |
|------|------|------|
| `NEXT_PUBLIC_LABOR_BASE_PATH` | 前端路由前缀 | `/labor` |
| `NEXT_PUBLIC_API_BASE` | 后端 API 根地址 | `https://host/labor` |
| `NEXT_PUBLIC_TENANT_ID` | 租户 ID（多租户场景） | `tenant-001` |
| `LABOR_API_PREFIX` | 后端 API 路由前缀 | `/labor` |

### 1.3 嵌入方式

- **iframe**：父应用通过 iframe 加载 `https://host/labor`，通过 postMessage 或 URL 参数传递租户 ID、主题等。
- **微前端（qiankun）**：将数字劳务作为子应用注册，通过 props 注入 `apiBase`、`tenantId`、`theme`。

## 二、模块化应用集成

### 2.1 插件注册表

前端提供 `web/lib/plugins.ts`：

- `registerPlugin(plugin)`：注册插件
- `getPlugins()` / `getPluginsByPermission(permissions)`：获取插件列表
- 插件包含：`id`、`name`、`entry`、`permission`、生命周期钩子

### 2.2 生命周期

- `beforeMount`：挂载前
- `mounted`：挂载后
- `beforeUnmount`：卸载前
- `unmounted`：卸载后

### 2.3 事件总线

- `on(event, handler)`：订阅
- `emit(event, payload)`：发布

标准事件：

- `contract:signed`：合同签署完成
- `settlement:confirmed`：结算确认完成
- `person:activated`：人员激活完成

---

## 三、API 对接规范

### 3.1 统一响应格式

- **成功**：`{ code: 0, message: "OK", data: T }`
- **失败**：`{ code: string, message: string, details?: any }`（HTTP 4xx/5xx）

### 3.2 分页

- 请求：`page`、`pageSize`
- 响应：`{ list, total, page, pageSize }`

### 3.3 OpenAPI

- FastAPI 自带 OpenAPI：`/docs`（Swagger UI）、`/openapi.json`
- 有 `LABOR_API_PREFIX` 时，文档路径为 `{prefix}/docs`、`{prefix}/openapi.json`

---

## 四、环境变量示例

```bash
# 嵌入场景
NEXT_PUBLIC_LABOR_BASE_PATH=/labor
NEXT_PUBLIC_API_BASE=https://api.example.com/labor
NEXT_PUBLIC_TENANT_ID=tenant-001
LABOR_API_PREFIX=/labor
```
